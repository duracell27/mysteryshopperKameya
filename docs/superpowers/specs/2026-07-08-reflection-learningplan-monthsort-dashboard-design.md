# Design: Reflection, Learning Plan Button, Monthly Sort, Dashboard Fix

**Date:** 2026-07-08  
**Scope:** 4 independent improvements to the Kameya mystery shopper app

---

## 1. Reflection — Hide Question 2 at 100%

### Problem
The reflection modal in `MyReportsView.tsx` always shows two questions:
1. "Що допомогло отримати такий результат?"
2. "Що я зроблю інакше наступного разу?"

For 100% scores, question 2 is not meaningful.

### Changes

**Frontend — `frontend/src/components/employee/MyReportsView.tsx`**
- When `selected.totalScore >= 100`, render only the first `<textarea>` in the reflection modal
- Skip `reflAnswer2` validation (min-length check) when score is 100%
- When submitting at 100%, send `answer2: ''` (empty string)

**Backend — `backend/src/routes/reports.ts` (POST `/:id/reflection`)**
- When `report.totalScore >= 100`, allow `answer2` to be empty
- Change validation from `if (!answer1?.trim() || !answer2?.trim())` to:
  - `if (!answer1?.trim())` → always required
  - `if (!answer2?.trim() && report.totalScore < 100)` → only required below 100%
- Store `answer2` as empty string `''` when not provided

## 2. Learning Plan Button in Employee Report View

### Problem
`MyReportsView.tsx` has no learning plan section. The plan can only be generated by admin. The backend already supports employee-triggered generation (`isOwner` check exists).

### UI Behavior (detail view, after the reflection block)

| Condition | UI |
|---|---|
| `totalScore === 100` | Disabled grey button + text: "Для анкет з 100% результатом проходити навчання не потрібно" |
| Reflection not yet submitted | Learning plan section hidden entirely |
| Reflection submitted, plan exists | Show plan tasks read-only (topic groups, completion checkmarks) |
| Reflection submitted, plan absent, score < 100 | Active "Згенерувати план навчання" button |

### Generation Flow (on button click)
1. If `report.aiRecommendations` is absent → call `POST /api/reports/:id/generate-ai`
2. Then call `POST /api/reports/:id/generate-learning-plan`
3. Show loading state during both calls; surface any error inline

### Admin Tracking
No backend changes needed. Admin already sees `learningPlan` in `AdminReportsListView.tsx`. Once the employee generates the plan, it appears in the admin view automatically.

### New State in `MyReportsView.tsx`
```
planLoading: boolean
planError: string | null
```
Plan data comes from `selected.learningPlan` (already on `AuditResult`).

### Plan Display (read-only)
Render task groups (by `topicTitle`) with check/cross icons and strikethrough text for completed tasks. No toggle interaction in this view — toggling tasks remains in `LearningPlanSection` on the Dashboard.

## 3. Monthly Grouping Toggle in Admin Reports List

### Problem
`AdminReportsListView.tsx` groups reports only by `quarter + year`. There is no way to see reports grouped by month.

### Changes — `frontend/src/components/admin/AdminReportsListView.tsx`

Add a two-option toggle above the search/store filters:
```
[ По кварталах ]  [ По місяцях ]
```
Default: "По кварталах" (existing behavior).

**Monthly grouping logic:**
- Group key: `${MONTHS_UK[report.month - 1]} ${report.year}` (e.g., "Липень 2025")
- Reports without `month` field → group under key `"Без місяця"`
- Sort groups: by year desc, then by month desc (reports without month last)

**State added:**
```
groupMode: 'quarter' | 'month'  (default: 'quarter')
```

No backend changes needed.

## 4. Dashboard Data Discrepancy Fix

### Problem
The `/stats/dashboard` endpoint filters reports using `{ year, month }`. Reports uploaded without the `month` field (it was always optional) do not appear in monthly dashboard views, causing count mismatches.

**Concrete examples from production:**
- 1 July report missing from July dashboard view (has `date: "2025-07-01"` but no `month: 7`)
- 12 June reports → only 10 shown (2 have no `month: 6`)

### Fix — `backend/src/routes/reports.ts` (GET `/stats/dashboard`)

Replace:
```typescript
if (periodType === 'month' && month) filter.month = month;
```

With a `$or` query that falls back to the `date` field when `month` is absent:
```typescript
if (periodType === 'month' && month) {
  const monthStr = String(month).padStart(2, '0');
  const datePrefix = `${year}-${monthStr}`;
  filter.$or = [
    { month },
    { month: { $exists: false }, date: { $regex: `^${datePrefix}` } },
    { month: null, date: { $regex: `^${datePrefix}` } },
  ];
}
```

**Why this is safe:** Admin-assigned `month` still takes priority. Only reports explicitly missing `month` fall back to the `date` field. A report assigned to a different month by the admin (e.g., `month: 6` for an audit dated July) will still appear in June, not July.

**Quarterly view:** No fix needed — `quarter` was always required at creation (`!quarter` → 400 error in confirm endpoint).
